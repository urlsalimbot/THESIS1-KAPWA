<action>
For each of these 7 page test files, add an axe a11y assertion. Process one page at a time: add the assertion → run the test → fix violations → commit.

**Setup:** vitest-axe is globally wired via `tests/setup.ts` (Phase 16-01). The `toHaveNoViolations()` matcher is available in every test without additional setup. Import `{ axe }` from `'vitest-axe'` at the top of each test file.

**Per-page procedure:**

1. **Add import:** At the top of the test file, add `import { axe } from 'vitest-axe';` (after existing vitest/testing imports, before the describe block).

2. **Add axe test:** Append a new `it()` block after the existing tests:
   ```tsx
   it('has no a11y violations', async () => {
     const { container } = render(<ExistingRenderSetup />);
     // Wait for async content if page uses useSWR
     await screen.findByRole('heading', { name: /Expected Heading/i });
     const results = await axe(container);
     expect(results).toHaveNoViolations();
   });
   ```

3. **Render wrapper selection** (match the page's existing test pattern):
   - **SWR pages** (AdminPage, BeneficiariesPage, CasesPage, CaseTrackerPage): Reuse the existing `renderWithSWR` helper that already returns container. Pattern: `const { container } = renderWithSWR(<Page />);`. Wait for heading via `await screen.findByRole(...)`.
   - **AccessCardPrintView** (route params): Use inline `render()` with MemoryRouter+Routes. Pattern: `const { container } = render(<MemoryRouter initialEntries={[...]}><Routes><Route .../></Routes></MemoryRouter>);`.
   - **AccessCardPage, ApprovalPipelinePage** — read each file to determine if it uses SWR or plain rendering. Use the same wrapper as existing tests.

4. **Run the test:** After adding the axe assertion, run:
   ```
   npx vitest run src/pages/<PageName>.test.tsx -t "has no a11y violations"
   ```

5. **If violations found:** Per D-07, fix the production code immediately:
   - Read the violation output: note the rule ID (e.g., `aria-required-children`, `heading-order`) and the CSS selector of the offending element.
   - Fix the production `.tsx` file (NOT the test file). Common fixes per RESEARCH.md Example 5:
     - Add `aria-label` to icon buttons
     - Fix heading hierarchy (no skipping h1→h3)
     - Add `aria-hidden="true"` to decorative icons
     - Add `<label htmlFor="...">` to form inputs
     - Add `role="presentation"` or `role="none"` to decorative elements
   - Re-run the test to confirm the fix resolves the violation.
   - If a violation cannot be fixed in scope (per D-09 — e.g., a shadcn primitive limitation), add a comment in the test file:
     `// axe-violation: <rule-id> — cannot fix because <reason>. Tracked in Phase 17 SUMMARY.`
     Do NOT use `skip` or `xit` — the test still runs (axe reports the violation, but it's acknowledged).

6. **Atomic commit:** After each page is green (0 violations or documented carve-out), commit:
   ```
   git add kapwa-client/src/pages/<PageName>.test.tsx [any production files changed]
   git commit -m "test(a11y): add axe assertion to <PageName>"
   ```
   If a production fix was needed, use:
   ```
   git commit -m "fix(a11y): resolve <violation-rule-id> in <PageName>"
   ```
   If multiple violations on one page required multiple production fixes, commit per fix.

Repeat for all 7 pages in this task. Run `npm run test:run` after all 7 pages are complete to ensure no regressions.
</action>
