## Spacing Scale

Declared values (inherited from Phase 7 design tokens — all multiples of 4):

| Token | Value | Usage in This Phase |
|-------|-------|---------------------|
| xs | 4px | Icon gaps inside buttons, card icon spacing |
| sm | 8px | Gap between nav links in header, card grid gaps (`gap-2`) |
| md (12) | 12px | Form field label-to-input gap |
| md (16) | 16px | Card padding (`p-4`), form field padding, section padding inside cards |
| lg | 24px | Section vertical spacing (`py-6` or `my-6`), hero padding |
| xl | 32px | Between major landing page sections (services → steps → about) |
| 2xl | 48px | Page-level top/bottom padding for hero, footer padding |
| 3xl | 64px | Maximum content width constraint (`max-w-4xl` = 896px), footer top padding |

Exceptions:
- **Touch targets on form controls (mobile):** All form inputs, buttons, select elements must have minimum 44px height on mobile viewports (`sm:` breakpoint and below). Use `h-11` (44px) or `py-2.5 min-h-[44px]` for form elements. This is a mobile-specific override; desktop can use standard `h-10` (40px). Source: WCAG 2.5.8 (Target Size minimum).
- **Hero section extra vertical padding:** `py-16 md:py-24` to create visual impact above the fold.

## Typography

Font families (inherited from Phase 7 design tokens):
- **Body:** Source Sans 3 (`--font-body`) — all body text, paragraphs, form labels, nav links
- **Heading:** Lexend (`--font-heading`) — all headings, hero title, section titles, brand text

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 16px | 400 (normal) | 1.5 | Landing page body paragraphs, service descriptions, step descriptions, contact info, form field values |
| Label | 14px | 500 (medium) | 1.25 | Form labels, nav link text, footer links, secondary metadata, step timeline numbers |
| Heading | 24px | 600 (semibold) | 1.2 | Section titles (Services, How to Apply, About Us), login/register page headings, about page mission heading |
| Display | 30px | 700 (bold) | 1.2 | Hero main headline, "KAPWA" brand text in header, page title on landing hero |

**Usage in this phase:**
- All Body text uses `font-body text-base leading-normal font-normal` (or equivalent Tailwind classes)
- All Label text uses `font-body text-sm leading-tight font-medium`
- All Heading text uses `font-heading text-2xl font-semibold leading-tight`
- All Display text uses `font-heading text-3xl font-bold leading-tight`
- Hero subheading: 18px (1.125rem) Body weight 400, line-height 1.5 — one step up from body for emphasis but not heading level
- Service card title: Heading (24px/600) on desktop, Body+ size (18px/600) on mobile
- Footer text: Label (14px/500) for links, Body (14px/400) for copyright line

**Weight summary:** 3 weights active — 400 (normal) for body text, 500 (medium) for labels/links, 600 (semibold) for headings, 700 (bold) for display/hero. The 700 weight is used exclusively for the hero headline to create maximum visual gravity.

**Accent reserved for:**
- Primary CTA buttons (`"Access Services"` hero button, `"Sign In"` submit, `"Create Account"` submit) — `Button variant="default"` with `bg-accent` (or default button which uses `bg-primary` — note: the current token setup has `primary` as dark/light foreground). Let me clarify: the `--accent` color should be used as background for the primary CTA buttons and submit buttons.
- Nav link hover and active states — `hover:text-accent`, the active/current page nav link uses `text-accent font-semibold`
- Body hyperlinks — `<a>` tags within content use `text-accent underline hover:text-accent/80`
- Form submit buttons — primary submit action (Sign In, Create Account, Send Message) uses accent as background
- Focus rings on all interactive elements — `ring-accent` or `focus-visible:ring-accent`
- **NOT** used for: decorative elements, card borders, section backgrounds, icons (use `text-muted-foreground` for icons)

**Note on Button variant mapping for this phase:**
- Primary CTA / form submit: `<Button className="bg-accent text-accent-foreground hover:bg-accent/90">` — explicitly uses accent color per user confirmation. Note: the default shadcn `Button variant="default"` maps to `--primary` which is dark navy; for this phase, primary action buttons use `--accent` (sky blue light, light blue dark).
- Secondary/outline actions: `<Button variant="outline">` for less emphasized actions (e.g., "Learn More" secondary cards)
- Ghost nav links: `<Button variant="ghost">` for header navigation items
- Link-style: `<Button variant="link">` for the "Register as claimant" entry below login form

### Login Button — Auth-Sensitive Toggle

On any public page (landing, about, contact), the header Login button dynamically toggles:

```tsx
{user ? (
  <Button variant="outline" asChild>
    <Link to={roleRedirectMap[user.role]}>Go to Dashboard</Link>
  </Button>
) : (
  <Button className="bg-accent text-accent-foreground hover:bg-accent/90" asChild>
    <Link to="/login">Login</Link>
  </Button>
)}
```

- Check auth state via `useAuth().user` or `getCurrentUser()` (from CONTEXT.md code context)
- No flash of wrong state: render nothing (`null`) until auth check completes (defer render until loading=false)

### Route Strategy

The public routes and the existing dashboard routes share `/`:

```
/login        → PublicLayout > LoginPage
/register     → PublicLayout > RegisterPage
/             → LandingPage (public) — redirects to /dashboard if authenticated
/about        → PublicLayout > AboutPage
/contact      → PublicLayout > ContactPage
/dashboard    → ProtectedRoute > Layout > DashboardPage  (new internal route)
/* (existing) → ProtectedRoute > Layout > [...existing dashboard pages]
```

**Important routing decision (the agent's discretion):**
- Current route at `/` renders `<Private><DashboardPage /></Private>`. The landing page also needs `/`.
- **Recommended approach:** Move the dashboard route to `/dashboard`. Set `/` to render `LandingPage` (public). Add a redirect: authenticated users visiting `/` automatically redirect to `/dashboard`. Unauthenticated visitors see landing page. This keeps the URL structure clean and avoids route conflicts.

**Redirect logic:**
```tsx
// In routes.tsx or a PublicRoute wrapper:
// If authenticated → <Navigate to="/dashboard" />
// If not authenticated → <LandingPage />
```

## Accessibility Requirements

| Requirement | Implementation |
|-------------|----------------|
| Skip-to-content | Inherited from Phase 8 Layout component. Used for dashboard pages. Public pages should also have skip-to-content: `<a href="#main-content">Skip to content</a>` as first focusable element in PublicLayout. |
| ARIA landmarks | `<header>` for PublicHeader, `<nav>` for header navigation, `<main id="main-content">` for page content, `<footer>` for PublicFooter. |
| Navigation landmark | `<nav aria-label="Main navigation">` for header links. Current page indicated with `aria-current="page"`. |
| Form labels | All form controls use `<Label>` component (shadcn) associated with `<Input>` via `htmlFor`/`id`. `<Label htmlFor="email">` + `<Input id="email">`. |
| Form errors | Inline error messages with `aria-describedby` on the input: `<Input aria-describedby="email-error" />` + `<p id="email-error" role="alert">`. |
| Form alerts | Error banner above login form uses `role="alert"`. MFA challenge container also uses `role="alert"`. |
| Toast notifications | Sonner toasts use `role="status"` and `aria-live="polite"` (built-in). |
| Interactive elements | All buttons are `<button>` or `<Button>` component (already semantic). Links use `<Link>` (React Router) rendered as `<a>`. |
| Focus management | Login form: auto-focus email input on render (`autoFocus`). MFA screen: auto-focus first digit input. Registration form: auto-focus name field. |
| Focus order (login page) | Header nav → Skip link → Logo → Email field → Password field → Sign In → Register link → Footer |
| Color contrast | All text colors use system tokens verified in Phase 7. Accent (#0369A1) on white (#FFFFFF) for CTA buttons: 4.6:1 contrast ratio — meets WCAG AA. |
| Touch targets | All interactive elements minimum 44×44px on mobile (WCAG 2.5.8). |
| Loading states | Buttons show spinner + text during submission. Skeleton components for any async-loaded content. |
